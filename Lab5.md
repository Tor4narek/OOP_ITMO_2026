# Лабораторная работа №5

## Архитектура приложения, SOLID и Dependency Injection

---

# Цель работы

Освоение проектирования архитектуры приложения на C# с использованием **SOLID**, слоистой архитектуры и **Dependency Injection**.

В рамках работы студент:

* проектирует архитектуру приложения перед реализацией;
* разделяет приложение на архитектурные слои;
* применяет принципы **SOLID**;
* внедряет зависимости через **Dependency Injection**;
* реализует сервисный слой приложения;
* использует **Entity Framework Core** в рамках архитектуры;
* применяет паттерны **Repository** и **Service Layer**.

# Общая идея приложения

* приложение реализуется как **чистый монолит**;
* код разделяется на **архитектурные слои**;
* зависимости между компонентами управляются через **Dependency Injection**;
* бизнес-логика не зависит от реализации хранения данных;
* приложение остается **консольным**.

# 1. Проектирование архитектуры

## Задача

Перед реализацией программы необходимо спроектировать архитектуру приложения.

## Требования

Приложение должно быть реализовано как **чистый монолит** с разделением на проекты.

Рекомендуемая структура solution:

```
Solution
├ Domain
├ Application
├ Infrastructure
└ Presentation.Console
```

---

## Назначение проектов

### Domain

Содержит:

* сущности предметной области;
* перечисления (`enum`);
* объекты-значения при необходимости;
* доменные исключения;
* бизнес-правила (инварианты).

Слой **не должен зависеть от других проектов**.

---

### Application

Содержит:

* прикладные сервисы (**use cases**);
* бизнес-логику приложения;
* сценарии работы системы;
* интерфейсы сервисов;
* интерфейсы репозиториев.

❗ **Не содержит кода EF Core**
❗ **Не использует DbContext напрямую**

---

### Infrastructure

Содержит:

* `DbContext`;
* конфигурацию Entity Framework Core;
* миграции;
* реализацию доступа к данным;
* реализации репозиториев.

❗ Реализует интерфейсы из `Application`

---

### Presentation.Console

Содержит:

* консольное меню;
* взаимодействие с пользователем;
* настройку Dependency Injection;
* точку входа в приложение.

❗ Не содержит бизнес-логики
❗ Не работает напрямую с базой данных

---

## Правило зависимостей

```
Presentation → Application
Infrastructure → Application
Application → Domain
Domain → none
```

❗ Нарушение этого правила считается архитектурной ошибкой

## UML-диаграмма архитектуры

Диаграмма должна отражать:

* проекты solution;
* зависимости между проектами;
* основные классы приложения;
* основные интерфейсы;
* сущности предметной области;
* прикладные сервисы;
* связи между компонентами системы.

❗ Диаграмма должна показывать **конкретные классы и интерфейсы**, а не только слои.

# 2. Применение принципов SOLID

## Задача

Спроектировать код таким образом, чтобы он соответствовал принципам **SOLID**.

В программе должны быть применены следующие принципы.

### Single Responsibility Principle

Каждый класс должен иметь **одну ответственность**

### Open / Closed Principle

Код должен быть открыт для расширения, но закрыт для изменения

### Liskov Substitution Principle

Наследники должны корректно заменять базовые типы

### Interface Segregation Principle

Интерфейсы должны быть небольшими и специализированными

### Dependency Inversion Principle

Зависимости должны строиться через абстракции

# 3. Service Layer

## Задача

Выделить **сервисный слой**, содержащий бизнес-логику приложения.

## Требования

Сервисы должны:

* реализовывать прикладные сценарии (**use cases**);
* координировать работу доменных сущностей и репозиториев;
* содержать бизнес-логику уровня приложения;
* вызываться из слоя представления.

---

## Обязательное требование

Необходимо реализовать:

* CRUD-операции;
* минимум **2 прикладных сценария (use case)** с бизнес-логикой.

Примеры:

* запись студента на курс;
* оформление заказа;
* выдача книги.

---

## Важно

❗ Сервис — это не просто CRUD-обертка над базой данных

---

# 4. Repository pattern

## Задача

Организовать доступ к данным через репозитории.

## Требования

* интерфейсы репозиториев размещаются в `Application`;
* реализации — в `Infrastructure`.

---

## Ответственность репозитория

Репозиторий отвечает только за:

* загрузку данных;
* сохранение данных;
* выполнение запросов.

❗ В репозитории **запрещено размещать бизнес-логику**

---

## Запрещено

❌ использовать универсальные generic репозитории (`IRepository<T>`)
❌ возвращать `IQueryable` наружу
❌ дублировать `DbSet<T>` без смысла

# 6. Dependency Injection

## Задача

Организовать внедрение зависимостей через контейнер DI.

## Требования

Использовать:

* Microsoft.Extensions.DependencyInjection

Регистрация зависимостей выполняется в `Presentation.Console`.

Пример:

```csharp
var services = new ServiceCollection();

services.AddDbContext<AppDbContext>();

services.AddScoped<IStudentRepository, StudentRepository>();
services.AddScoped<StudentService>();

var provider = services.BuildServiceProvider();
```

---

## Правила

* зависимости внедряются через конструкторы;
* создание зависимостей через `new` запрещено.

---

## Lifetime

* Transient — новый объект каждый раз
* Scoped — один объект на операцию
* Singleton — один объект на приложение

❗ Студент должен понимать различия
